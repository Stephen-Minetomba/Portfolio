LLSPL

The Definitive Guide to the Low Level Simple Programming Language

1. Design Philosophy

LLSPL (Low Level Simple Programming Language) was born from a desire for absolute transparency in systems development. In an era where "modern" languages hide complexity behind layers of abstraction, LLSPL moves in the opposite direction. It is a language for those who want to see the gears turning—for those who believe that understanding the machine is the first step to mastering it.

The core tenets of LLSPL are:

  • Transparency: Every line of code should correspond to a predictable sequence of machine instructions.
  • Efficiency: No hidden heap allocations, no garbage collection pauses, and no bloated runtime.
  • Control: Direct access to hardware registers, memory-mapped I/O, and CPU-specific instructions.
  • Portability: While currently targeting x86-64, the architecture of the transpiler allows for future backends targeting RISC-V or ARM.

2. Project Architecture

The LLSPL ecosystem is a tightly integrated suite of tools designed to take you from a high-level concept to raw machine execution.

4. Language Syntax

LLSPL syntax is influenced by C and Go, focusing on readability and simplicity.

4.1 Variable Declaration

Variables are declared with a type, an identifier, and an optional initializer.

i64 x = 100;
u32 flags; // Defaults to 0
i8 message[128]; // Stack-allocated array

4.2 Function Definition

i64 calculate_sum(i64 a, i64 b) {
    i64 result = a + b;
    return result;
}

4.3 Control Flow

if (x > 10) {
    x = x - 1;
} else {
    x = x + 1;
}

while (x > 0) {
    process(x);
    x = x - 1;
}

5. Pointers & Memory

In LLSPL, pointers are first-class citizens. They are simply 64-bit unsigned integers that represent memory addresses.

i64 x = 42;
i64* ptr = &x; // Address-of operator
*ptr = 100;    // Dereference and assign
Pointer arithmetic is supported. Adding 1 to a pointer increases the address by sizeof(type).

6. Data Structures

Structs are used to group related data. The compiler ensures zero-padding unless specific alignment is requested.

struct Vector3 {
    i64 x;
    i64 y;
    i64 z;
};

struct Vector3 v;
v.x = 10;
v.y = 20;
v.z = 30;

7. Inline Assembly

When you need to go truly "raw", the asm block allows for direct insertion of NASM code.

void pause() {
    asm {
        hlt
    }
}

void outb(u16 port, u8 val) {
    asm {
        mov dx, [port]
        mov al, [val]
        out dx, al
    }
}

8. Memory Model

LLSPL uses a Static Allocation Model. Every variable defined in a function is assigned a static label in the .data section.

This means recursion is not natively supported without a custom stack implementation.

9. UEFI Integration

LLSPL is uniquely optimized for UEFI. The efi_setup block handles the complexity of the transition from bootloader to OS.

efi_setup {
    initialize_display();
    main();
}

%macro getIndexR 4 ; dest, array, index_reg, bits
    mov rax, [%2]
    mov rbx, [%3]

    ; auto compute scaling: bits / 8
    %assign SCALE %4 / 8
    imul rbx, SCALE
    add rax, rbx

    %ifidn %4, 8
        mov bl, [rax]
        mov [%1], bl
    %elifidn %4, 16
        mov bx, [rax]
        mov [%1], bx
    %elifidn %4, 32
        mov ebx, [rax]
        mov [%1], ebx
    %elifidn %4, 64
        mov rbx, [rax]
        mov [%1], rbx
    %else
        %error "Invalid bit width for getIndexR (must be 8,16,32,64)"
    %endif
%endmacro

    push rbx
    push rbp
    mov rbp, rsp
    sub rsp, 128
    mov [rbp - 40], rdi
    mov [rbp - 48], rsi
.exit_bservices_retry:
    mov rsi, [rbp - 48]
    mov rbx, [rsi + 96]
    mov qword [rbp - 8], 0
    lea rcx, [rbp - 8]
    xor rdx, rdx
    lea r8, [rbp - 16]
    lea r9, [rbp - 24]
    lea rax, [rbp - 32]
    mov [rsp + 32], rax
    mov rax, [rbx + 56]
    call rax
    mov rcx, [rbp - 40]
    mov rdx, [rbp - 16]
    mov rax, [rbx + 232]
    call rax
    test rax, rax
    jnz .exit_bservices_retry
    mov rsp, rbp
    pop rbp
    pop rbx
%endmacro

    mov rdi, rcx
    mov rsi, rdx
    EXIT_BOOT_SERVICES
    mov rsp, kernel_stack + 4096
%endmacro

11. Compiler Internals

The LLSPL compiler is a 4-stage pipeline.

Stage 1: Lexical Analysis

The tokenize function uses Python's re module to match strings against regular expressions for IDs, Keywords, and Operators.

Stage 2: Parsing

The Parser class implements a recursive descent algorithm. It maintains a pos pointer to the token stream and builds an AST (Abstract Syntax Tree).

Stage 3: Optimization

Constant folding and dead code elimination are performed on the AST before code generation.

Stage 4: Code Generation

The X86Transpiler walks the AST and emits NASM assembly strings, utilizing the macros defined in lib.asm.

12. Advanced Tutorials

Tutorial A: Writing a CMOS Clock Reader

Accessing the CMOS to read the system time via ports 0x70 and 0x71.

u8 read_cmos(u8 reg) {
    u8 val;
    asm {
        mov al, [reg]
        out 0x70, al
        in al, 0x71
        mov [val], al
    }
    return val;
}

Tutorial B: Implementing a Spinlock

Using atomic instructions for synchronization.

void lock(u64* mutex) {
    asm {
        .retry:
            lock bts qword [mutex], 0
            jc .retry
    }
}

void unlock(u64* mutex) {
    *mutex = 0;
}

13. Formal Grammar (EBNF)

program         = { struct_def | var_decl | func_def | efi_setup | asm_block } ;
struct_def      = "struct" identifier "{" { type identifier ";" } "}" [ ";" ] ;
func_def        = type identifier "(" [ param_list ] ")" "{" { statement } "}" ;
param_list      = type identifier { "," type identifier } ;
statement       = var_decl | assignment | if_stmt | while_stmt | return_stmt | call_stmt | asm_block | label_def | goto_stmt ;
assignment      = expression "=" expression ";" ;
if_stmt         = "if" "(" expression ")" "{" { statement } "}" [ "else" "{" { statement } "}" ] ;
while_stmt      = "while" "(" expression ")" "{" { statement } "}" ;
return_stmt     = "return" [ expression ] ";" ;
goto_stmt       = "goto" identifier ";" ;
label_def       = identifier ":" ;
expression      = term { ( "||" | "&&" | "==" | "!=" | "<" | "<=" | ">" | ">=" | "+" | "-" ) term } ;
term            = factor { ( "*" | "/" | "%" ) factor } ;
factor          = number | identifier | "(" expression ")" | call_expr | array_access | member_access | deref_expr | addr_expr | sizeof_expr ;
type            = ( "i8" | "u8" | "i16" | "u16" | "i32" | "u32" | "i64" | "u64" | "u" | "void" | "struct" identifier ) { "*" } ;

14. Detailed FAQ

Q01: Why is it called LLSPL?
It stands for Low Level Simple Programming Language. The name reflects its goal: providing a simple, high-level-ish interface to low-level hardware.
Q02: How do I handle multi-file projects?
Currently, you must concatenate your .c files or use %include in the generated assembly. A linker-based approach is planned for v2.0.
Q03: Does it support floating point?
Not natively. You can use asm blocks to interact with the XMM registers and the FPU.
Q04: How can I debug my LLSPL code?
The best way is to use QEMU with GDB. You can set breakpoints in the generated assembly and step through the instructions.
Q05: What is the maximum size of a struct?
Technically unlimited, restricted only by the available memory of your target machine.
Q06: Can I use LLSPL to write a Windows driver?
While possible (as it outputs x86-64), LLSPL is better suited for freestanding environments like UEFI or custom kernels.
Q07: Why are all variables static?
To eliminate the complexity of stack management and frame pointers, making the compiler easier to audit and the code faster to execute in minimal environments.
Q08: How do I pass a struct by value?
The compiler handles this by using rep movsb to copy the struct data into the destination's static memory area.
Q09: Is there a `sizeof` operator?
Yes. sizeof(type) or sizeof(variable) returns the size in bytes.
Q10: Can I declare local variables inside a `while` loop?
Yes, but remember they are static and shared across all loop iterations.

15. Systems Glossary

Alignment

The requirement that data addresses be multiples of the data size (e.g., a 64-bit value should be at an address divisible by 8).

BCD (Binary Coded Decimal)

A class of binary encodings of decimal numbers where each digit is represented by a fixed number of bits.

CPUID

An x86 instruction that returns information about the processor's features and identification.

DLAB

Divisor Latch Access Bit, used in UART communication to set the baud rate.

EFLAGS

A status register in x86 processors that contains the current state of the CPU.

Freestanding

An environment where no operating system services or standard libraries are available.

GDT (Global Descriptor Table)

A data structure used by x86 processors to define the characteristics of memory segments.

IDT (Interrupt Descriptor Table)

A data structure used by x86 processors to implement an interrupt vector table.

Kernel

The core part of an operating system that manages system resources and hardware communication.

Little-endian

A byte ordering where the least significant byte is stored at the lowest address (used by x86).

MMIO (Memory Mapped I/O)

A method of performing I/O between the CPU and peripheral devices using the same address space as RAM.

Paging

A memory management scheme by which a computer stores and retrieves data from secondary storage for use in main memory.

Real Mode

The 16-bit operating mode of x86 processors characterized by a 20-bit segmented memory address space.

SSE (Streaming SIMD Extensions)

A SIMD instruction set extension to the x86 architecture.

TSS (Task State Segment)

A special segment on x86 processors used to store information about a task.

VGA (Video Graphics Array)

A legacy video standard and hardware interface used for text and basic graphics display.

Z-buffer

A method in 3D computer graphics used to manage image depth coordinates.

16. Troubleshooting

Common issues and their resolutions.

Symptom Probable Cause Resolution
Segmentation Fault Null pointer dereference or out-of-bounds access. Check your pointer values and array indices.
Stack Overflow (in UEFI) Too many nested function calls. Increase the stack size in efi_setup.
Undefined Symbol Missing macro or function declaration. Ensure lib.asm is included and function names match.
